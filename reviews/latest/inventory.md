## Module 1 ‚Äî gaviero-core
Path: crates/gaviero-core
Purpose: The runtime heart of the workspace. Hosts all non-UI logic: multi-agent swarm orchestration (tier routing, parallel fan-out, dependency DAG, verification gates, merge, replan, calibration, context bundle assembly), the multi-database ONNX-embedding semantic memory subsystem with three-cadence consolidation, an in-process read-only MCP server (`memory_search`, `blast_radius`, `node_doc`) reachable by subprocess agents over `<workspace>/.gaviero/mcp.sock`, the legacy ACP Claude subprocess transport plus a backend-neutral `agent_session` transport (claude / codex_exec / codex_app_server / ollama), the interactive Write Gate (file-modification boundary with diff review and scope enforcement), validation gates, scope/path-pattern checks, a tree-sitter registry for 16 languages, repo-map knowledge graph, unified diff engine, `git2`-backed git/worktree wrapper, PTY-driven embedded terminal, and shared boundary types. No UI or DSL dependencies ‚Äî TUI/CLI/DSL all consume this crate.
Languages: Rust
Key entry points: crates/gaviero-core/src/lib.rs, crates/gaviero-core/src/swarm/mod.rs, crates/gaviero-core/src/swarm/pipeline.rs, crates/gaviero-core/src/swarm/coordinator.rs, crates/gaviero-core/src/swarm/backend/mod.rs, crates/gaviero-core/src/memory/services.rs, crates/gaviero-core/src/mcp/, crates/gaviero-core/src/acp/mod.rs, crates/gaviero-core/src/write_gate/, crates/gaviero-core/src/path_pattern.rs

## Module 2 ‚Äî gaviero-tui
Path: crates/gaviero-tui
Purpose: Full-screen terminal editor (`gaviero` binary). Pure rendering and input layer over `gaviero-core` ‚Äî every event channel sender on the `App` implements core's observer traits (`WriteGateObserver`, `AcpObserver`, `SwarmObserver`) so background work is funnelled into a single `mpsc` event queue and processed on a `draw ‚Üí recv ‚Üí handle ‚Üí repeat` main loop. Provides the file-tree, editor (Ropey buffer + diff overlay + tree-sitter-driven syntax highlighting), agent-chat panel with chat-memory bridge, swarm dashboard, git panel, search panel, memory inspection panel, status bar, and embedded PTY terminal. Holds layout, focus, keymap (Ctrl = editor, Alt = workspace layering), theming, and session restore ‚Äî but never mutates app state from a background task.
Languages: Rust
Key entry points: crates/gaviero-tui/src/main.rs, crates/gaviero-tui/src/app.rs, crates/gaviero-tui/src/app/, crates/gaviero-tui/src/event.rs, crates/gaviero-tui/src/keymap.rs, crates/gaviero-tui/src/editor/, crates/gaviero-tui/src/panels/

## Module 3 ‚Äî gaviero-cli
Path: crates/gaviero-cli
Purpose: Headless command-line runner (`gaviero-cli` binary) for CI and scripting. A thin clap wrapper that wires stderr observers around the same `gaviero-core` pipelines as the TUI and consumes `gaviero-dsl` for `.gaviero` workflow scripts. Exposes four execution modes ‚Äî `--task` (single-agent full-repo), `--work-units` (manual JSON `WorkUnit` array), `--script` (compile and execute a DSL file with optional `--workflow` selector and `--var KEY=VALUE` overrides), and `--coordinated` (Opus plans a DAG, emits a `.gaviero` for review, then exits) ‚Äî plus repo-map graph build (`--graph`), memory namespace plumbing (`--namespace`/`--read-ns`/`--no-memory`), retry/iteration controls (`--max-retries`, `--attempts`, `--test-first`, `--no-iterate`), structured trace output (`--trace-log`), and resume from `.gaviero/state/<plan-hash>.json` (`--resume`). The `Cli` clap struct in `src/main.rs` is the authoritative flag list.
Languages: Rust
Key entry points: crates/gaviero-cli/src/main.rs

## Module 4 ‚Äî gaviero-dsl
Path: crates/gaviero-dsl
Purpose: Compiler for `.gaviero` workflow scripts. A four-stage pipeline ‚Äî Logos lexer ‚Üí Chumsky parser ‚Üí AST (`Script`, `Item`, `AgentDecl`, `WorkflowDecl`, `PromptDecl`, `TierAlias`) ‚Üí semantic analyser/compiler that produces a `CompiledPlan` DAG consumable by `gaviero-core`'s swarm pipeline. Validates scope-glob overlaps via `gaviero_core::path_pattern::paths_overlap` (disjoint globs allowed; overlap within a single `loop { agents [...] }` group permitted) and dependency cycles at compile time. Top-level surface includes `client`, `agent`, `workflow`, `prompt`, `vars`, and `tier <alias> <client-ref>`. Public API: `compile(...)` and `compile_with_vars(...)`, the latter backing `gaviero-cli --var KEY=VALUE` with precedence agent-level `vars {}` > CLI `--var` > script-level `vars {}`. Errors are Miette diagnostics with source spans.
Languages: Rust
Key entry points: crates/gaviero-dsl/src/lib.rs, crates/gaviero-dsl/src/lexer.rs, crates/gaviero-dsl/src/parser.rs, crates/gaviero-dsl/src/ast.rs, crates/gaviero-dsl/src/compiler.rs, crates/gaviero-dsl/src/error.rs, crates/gaviero-dsl/src/resolver.rs

## Module 5 ‚Äî gaviero-mcp-shim
Path: crates/gaviero-mcp-shim
Purpose: Self-contained `gaviero-mcp-shim` binary that bridges stdio (the protocol subprocess coding agents like Claude Code and Codex speak) to `gaviero-core`'s in-process MCP server over a Unix domain socket at `<workspace>/.gaviero/mcp.sock`. This indirection is what lets agents that cannot embed `gaviero-core` directly still reach its read-only `memory_search` / `blast_radius` / `node_doc` tools. Has zero workspace dependencies (only `tokio`, `clap`, `anyhow`, `tracing`, `tracing-subscriber`) so it can ship as an independent executable; `gaviero-core::mcp::config_synth` writes the `.mcp.json` entries that point agents at this binary.
Languages: Rust
Key entry points: crates/gaviero-mcp-shim/src/main.rs

## Module 6 ‚Äî tree-sitter-gaviero
Path: crates/tree-sitter-gaviero
Purpose: Tree-sitter grammar for `.gaviero` files, providing the incremental syntax tree consumed by the editor's syntax-highlighting overlay and any tooling that walks DSL source. `grammar.js` is the hand-maintained source of truth; `parser.c`, `grammar.json`, and `node-types.json` are generated artefacts (regenerated via the tree-sitter CLI and committed to the repo). `build.rs` compiles the generated C parser via the `cc` crate, and `src/lib.rs` exposes the Rust bindings (the `LANGUAGE` constant) plus integration tests that exercise the grammar against representative DSL snippets. Strict scope: syntax only ‚Äî semantic analysis, name resolution, and validation belong to `gaviero-dsl`. Downstream code must consume this grammar through the re-export in `gaviero-core::lib.rs` rather than depending on the `tree-sitter` crate directly.
Languages: JavaScript (grammar source), C (generated parser), Rust (bindings + tests)
Key entry points: crates/tree-sitter-gaviero/grammar.js, crates/tree-sitter-gaviero/build.rs, crates/tree-sitter-gaviero/src/lib.rs, crates/tree-sitter-gaviero/src/parser.c, crates/tree-sitter-gaviero/src/grammar.json, crates/tree-sitter-gaviero/src/node-types.json

## Global assessment

- **Architecture**: Strict layered Cargo workspace with a single hub-and-spokes shape. `gaviero-core` is the source of truth for runtime logic and pulls in zero workspace siblings. `tree-sitter-gaviero` is a leaf grammar consumed only by `gaviero-core` (and re-exported from there to avoid downstream `tree-sitter` imports). `gaviero-dsl` depends on `gaviero-core` for `CompiledPlan` and `path_pattern`. The two binary frontends ‚Äî `gaviero-tui` and `gaviero-cli` ‚Äî both depend on `gaviero-core` + `gaviero-dsl` and on nothing else from the workspace; they never depend on each other. `gaviero-mcp-shim` is deliberately detached: zero workspace deps, so it ships as a portable bridge for subprocess agents. Dependency direction is unidirectional (UI/CLI ‚Üí DSL ‚Üí core ‚Üê grammar; shim is orthogonal). No cycles, no layering violations observed.

- **Boundaries**: Public APIs are well-defined and largely respected. `gaviero-core::lib.rs` enumerates 22 public modules and is the documented surface for downstream crates. The TUI's `CLAUDE.md` explicitly forbids background tasks from mutating `App` directly ‚Äî all cross-crate communication funnels through the observer traits (`WriteGateObserver`, `AcpObserver`, `SwarmObserver`) into a single event channel, which is a clean inversion-of-control pattern. The DSL compiler reuses `gaviero_core::path_pattern` rather than reimplementing scope overlap detection. Tree-sitter is exposed exclusively through `gaviero-core`'s re-export to keep the version pinned in one place. The MCP server is read-only by construction (the `gaviero-core/CLAUDE.md` calls this an invariant ‚Äî "Never add a write tool there"), with all writes routed through the Write Gate or a single-consumer memory writer mpsc task. The shim's stdio‚Üîsocket bridge is the documented boundary between subprocess agents and core's in-process MCP server.

- **Cross-cutting concerns**: Highly consistent across the Rust crates. Error handling is `anyhow::Result` everywhere except `gaviero-dsl`, which uses `thiserror` + Miette diagnostics for span-aware user-facing errors (justified by the compile-time-error use case). Logging is uniformly `tracing` with `tracing-subscriber` at binary entry points. Async is uniformly Tokio. Boundary types use `serde` derives. Workspace-level dependency declarations in the root `Cargo.toml` keep versions of `tokio`, `serde`, `anyhow`, `tracing`, `clap`, `reqwest`, `logos`, `chumsky`, `miette`, `thiserror`, and `dirs` aligned across crates. Lock discipline is documented as an invariant: never hold a Mutex across I/O, parsing, or embedding (memory writes are single-consumer through an mpsc writer task, embedding runs outside the SQLite Mutex). `git2` is mandated ‚Äî no shelling out to `git`. Edition is `2024` for every Rust crate except `tree-sitter-gaviero` (`2021`), a minor inconsistency likely chosen to track upstream tree-sitter binding conventions.

- **Hot spots**: `gaviero-core` is by far the largest and most active surface ‚Äî 22 public modules covering swarm orchestration, memory, MCP, write gate, agent transport, repo-map, terminal emulation, and tree-sitter integration; the swarm subsystem alone has ~30 files (pipeline, coordinator, planner, replanner, validation, calibration, router, merge, board, bus, execution_state, plan, privacy, context, context_bundle, ollama, plus a `backend/` directory with claude_code/codex/ollama/mock backends behind a unified `AgentBackend` trait, and a `verify/` directory for structural/diff-review/test-runner/combined verification). Memory is similarly dense, with three-cadence consolidation (extractor ‚Üí session_consolidator ‚Üí sleeptime), pluggable embedders (`gte-modernbert-base` default, `nomic-embed-text-v1.5` legacy), a multi-DB writer-task architecture, and a recently-introduced merged multi-scope RRF retrieval default with the legacy cascade kept as a kill-switch. `gaviero-tui` is the second-largest crate with rich panel coverage and an embedded PTY. `gaviero-cli` is intentionally minimal ‚Äî a single `~2.1 KLOC` `main.rs` whose `Cli` clap struct is the authoritative flag list. `gaviero-dsl` is small and stable but central to the coordinated-mode workflow. `tree-sitter-gaviero` is the most stable module (mostly generated, hand edits restricted to `grammar.js`). `gaviero-mcp-shim` is a small bridge utility that should rarely change once protocol-stable. None of the modules look stale; the `git status` snapshot shows in-flight work concentrated in the DSL (lexer/parser/compiler/AST/error/resolver), DSL examples, and the tree-sitter grammar (`grammar.js` + regenerated artefacts) ‚Äî consistent with an active DSL surface evolution.
