# gaviero-dsl

`gaviero-dsl` is the declarative workflow language for Gaviero. A `.gaviero`
file describes clients, agents, scopes, memory/context hints, workflow
iteration, verification, and explicit loops. The compiler turns that source
into `gaviero_core::swarm::plan::CompiledPlan`.

The same format is also used for reviewable coordinated plans generated by the
CLI and TUI before execution.

## Core concepts

- `client`: model/tier/privacy defaults for agents
- `agent`: a unit of work with scope, prompt, dependencies, and optional
  memory/context blocks
- `workflow`: execution order, parallelism, iteration strategy, verification,
  and loop rules

## Quick example

```gaviero
client reasoning {
    tier  expensive
    model "sonnet"
}

client local_exec {
    tier    cheap
    model   "ollama:qwen2.5-coder:7b"
    privacy local_only
}

agent design {
    description "Plan the refactor"
    client reasoning
    scope {
        read_only ["src/" "tests/" "docs/"]
    }
    prompt #"
        Inspect the current implementation and produce a concrete refactor plan.
        Call out risks, missing tests, and file ownership.
    "#
}

agent implement {
    description "Apply the refactor"
    client local_exec
    depends_on [design]
    scope {
        owned        ["src/" "tests/"]
        read_only    ["docs/"]
        impact_scope true
    }
    context {
        callers_of ["src/auth/session.rs"]
        tests_for  ["src/auth/"]
        depth      2
    }
    prompt "Implement the approved plan and keep changes scoped."
    max_retries 4
}

workflow refactor_auth {
    steps [design implement]
    strategy refine
    verify {
        compile      true
        clippy       true
        impact_tests true
    }
}
```

## Model strings

The DSL does not impose provider-specific enums on `model`. It stores the
string and lets `gaviero-core` resolve it later.

- `sonnet`, `opus`, `haiku`: Claude by default
- `claude:<name>` / `claude-code:<name>`: explicit Claude selection
- `ollama:<model>` / `local:<model>`: local Ollama model selection

## Language features

### Scopes

Use `scope {}` to declare ownership and read boundaries.

- `owned [...]`: files/directories the agent may modify
- `read_only [...]`: extra readable paths
- `impact_scope true`: expand context using the code graph around owned files

### Graph context

Use `context {}` when the agent needs graph-derived context.

- `callers_of [...]`
- `tests_for [...]`
- `depth <n>`

### Memory

Use `memory {}` to read from or write to named namespaces and to control
prompt-enrichment behavior.

### Workflow iteration

`workflow` supports:

- `strategy single_pass | refine | best_of_N`
- `max_retries`
- `attempts`
- `test_first`
- `escalate_after`

### Verification

`verify {}` currently supports:

- `compile true`
- `clippy true`
- `test true`
- `impact_tests true`

### Explicit loops

Workflows can include `loop {}` blocks with `max_iterations` and one of three
exit conditions:

- verification-based `until { ... }`
- judge-agent `until agent reviewer`
- shell-command `until command "cargo test"`

## Using the DSL

### From the TUI

- `/run path/to/workflow.gaviero`
- `/run path/to/workflow.gaviero runtime prompt text`

The second form passes a runtime prompt into the compiler, which can be
substituted with `{{PROMPT}}` inside prompt strings.

### From the CLI

- `gaviero-cli --script path/to/workflow.gaviero`

The CLI currently compiles scripts as-is; it does not inject a separate runtime
prompt into `--script` mode.

### As coordinated output

`/cswarm ...` in the TUI and `gaviero-cli --coordinated --task ...` generate a
reviewable `.gaviero` plan that you can inspect, edit, and then execute.

## Scope of this crate

`gaviero-dsl` is syntax and compilation only. Runtime execution, provider
resolution, verification, and merge behavior all live in `gaviero-core`.
